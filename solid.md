# SOLID 원칙 정리

## 1. SRP (Single Responsibility Principle)

### 개념
하나의 클래스는 하나의 책임만 가져야 한다.  
즉, 클래스가 변경되는 이유는 하나여야 한다.

### 핵심
- 책임이 많아질수록 코드 변경 시 영향 범위가 커짐
- 유지보수 어려움 증가

### 예시
- 사용자 정보 관리 + 로그인 기능을 한 클래스에 넣는 것 → 잘못된 설계
- 사용자 정보 관리 / 로그인 기능을 각각 분리 → 올바른 설계

---

## 2. OCP (Open-Closed Principle)

### 개념
확장에는 열려 있고, 수정에는 닫혀 있어야 한다.

### 핵심
- 기존 코드를 수정하지 않고 기능을 추가할 수 있어야 함
- 다형성과 인터페이스를 활용

### 예시
- 새로운 기능 추가 시 기존 코드 수정 → 위반
- 인터페이스 구현체 추가로 기능 확장 → 준수

---

## 3. LSP (Liskov Substitution Principle)

### 개념
자식 클래스는 부모 클래스를 대체할 수 있어야 한다.

### 핵심
- 부모 타입을 사용하는 곳에 자식 객체를 넣어도 문제 없어야 함
- 기능의 의미가 변하면 안 됨

### 예시
- 부모: “증가” 기능
- 자식: 실제로 감소 → 원칙 위반

---

## 4. ISP (Interface Segregation Principle)

### 개념
인터페이스는 클라이언트에 맞게 분리해야 한다.

### 핵심
- 하나의 큰 인터페이스보다 여러 개의 작은 인터페이스가 좋음
- 불필요한 기능 의존 제거

### 예시
- 하나의 인터페이스에 모든 기능 정의 → 비효율
- 기능별로 인터페이스 분리 → 효율적

---

## 5. DIP (Dependency Inversion Principle)

### 개념
구체적인 클래스가 아닌 추상화(인터페이스)에 의존해야 한다.

### 핵심
- 구현이 아닌 인터페이스에 의존
- 의존성 주입(DI)을 통해 구현

### 예시
- 구체 클래스 직접 사용 → 변경에 취약
- 인터페이스를 통해 의존 → 유연성 증가

---

## 전체 정리

- SRP: 하나의 책임만 가져라
- OCP: 수정 없이 확장 가능하게 만들어라
- LSP: 자식은 부모를 대체할 수 있어야 한다
- ISP: 인터페이스는 작게 나눠라
- DIP: 추상화에 의존하라
